# Assumption Audit

Use this before committing to a design or a study. It turns vague confidence into a ranked list of what you actually need to learn.

## For each assumption

| Field | Your notes |
| --- | --- |
| We believe… | |
| We believe it because… | |
| Evidence we have (and its strength: observed / reported / stated) | |
| Impact if we're wrong (low / medium / high) | |
| What we'd need to learn | |
| The decision this informs | |

## Prioritize

Plot each assumption on two axes and test the top-right first.

- **Impact if wrong:** how badly does the design break?
- **Current evidence:** how much do we actually know?

> Test the high-impact, low-evidence assumptions first. The rest can wait.

## Quality check

- [ ] Each assumption names a decision it informs.
- [ ] Evidence is labeled by strength (observed beats reported beats stated).
- [ ] At least one assumption could realistically be proven wrong.
- [ ] The riskiest assumption is identified and has a plan to test it.

## Worked example (healthcare appointment portal)

| Field | Example |
| --- | --- |
| We believe… | Patients abandon booking because the flow is too long. |
| Because… | The flow has eight steps and the team finds it tedious. |
| Evidence (strength) | None observed — stated team opinion only. |
| Impact if wrong | High — we'd shorten the wrong thing. |
| What we'd learn | Where and why patients actually drop out. |
| Decision it informs | Whether to cut steps or restructure (e.g. defer insurance). |

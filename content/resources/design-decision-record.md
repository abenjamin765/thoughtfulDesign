# Design Decision Record

Use one per significant decision. Short enough to actually write; complete enough to actually help a year from now. This is how judgment compounds instead of evaporating.

## The record

| Field | Your entry |
| --- | --- |
| Decision | |
| Date / owner | |
| Context (what prompted it) | |
| User need | |
| Business goal | |
| Evidence (and strength) | |
| Tradeoff accepted | |
| Main alternative & why it lost | |
| Revisit trigger (what would reopen this) | |

## Why it matters

- It lets others evaluate your reasoning, not just your pixels.
- It survives turnover and time, so the decision isn't blindly reversed.
- It closes the loop: when the revisit trigger fires, you Adapt deliberately.

## Quality check

- [ ] Captures the *why*, not just the choice.
- [ ] Names a real tradeoff and the losing alternative.
- [ ] States a concrete revisit trigger.
- [ ] Short enough that you'll actually keep doing it.

## Worked example

> **Decision:** Recommendations are reviewable suggestions teachers approve, not auto-applied.
> **User need:** Teachers distrust recommendations built on uneven data and override them.
> **Business goal:** Reduce planning time and increase engagement.
> **Tradeoff:** Less dramatic time savings than full automation.
> **Alternative:** Auto-apply — rejected because wrong recommendations would reach students unreviewed.
> **Revisit trigger:** If data quality becomes consistently high and teacher trust is established, reconsider partial automation.

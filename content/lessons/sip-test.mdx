## A filter for real objects

Not every noun in a requirements doc deserves object status. The **SIP test** helps teams separate true objects from attributes, actions, or implementation details.

- **S**tructure — Does it have a recognizable shape users can point to?
- **I**dentity — Can users distinguish one instance from another?
- **P**urpose — Does it exist for a clear user-facing reason?

<Callout type="tip">

Run SIP early during object discovery. It prevents object sprawl and keeps your model aligned to user mental models.

</Callout>

## Worked example: LIBRARY catalog

| Candidate | SIP result |
| --- | --- |
| BOOK | Pass — users browse, identify, and care about books |
| ISBN | Fail — attribute of BOOK, not its own object |
| "Search" | Fail — action, not an object |
| LIBRARIAN | Pass — a person users interact with as a distinct entity |

<Callout type="warning">

Passing SIP in a workshop does not mean the object belongs in v1. Prioritization comes later — SIP only validates object candidacy.

</Callout>

## Try it on your domain

Pick three nouns from a current project. For each one, ask:

1. Can a user point to it in the interface?
2. Would they describe two instances as different versions of the same thing?
3. Would they miss it if it disappeared?

<Reveal label="SIP shorthand">

If it fails Purpose, it might be a feature. If it fails Identity, it might be metadata. If it fails Structure, it might be a relationship in disguise.

</Reveal>

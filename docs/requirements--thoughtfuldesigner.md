# Thoughtful Design Course Requirements

## 1. Course Overview

**Course title:** Thoughtful Design
**Working subtitle:** A modern UX practice for designing complex, intelligent, human-centered systems.

**Course format:** Self-led, asynchronous, text-based digital course.

**Platform architecture:** The platform uses a two-zone structure selected in Phase 5 concept evaluation ("The Structured Program," Concept B). The **public marketing zone** — landing page, blog, resources library, and tenets manifesto section — requires no authentication and serves discovery, trust-building, and organic search. The **authenticated learning zone** — dashboard, course overview, module pages, lessons, assessments, and certificate — is gated behind a learner account. This separation keeps the enrollment path clear and the learning experience distraction-free. One lesson per module is made publicly accessible (no login required) as a preview, showing the core claim and one example to support SEO and visitor trust without blurring the course/blog boundary.

**Primary audience:**
UX designers, product designers, UX researchers, design-adjacent product professionals, and early-to-mid career practitioners who want to move beyond screen-level execution into systems-level design judgment.

**Course purpose:**
Thoughtful Design prepares learners for the next generation of UX work by teaching them to understand systems, model product domains, conduct research-informed design, facilitate better decisions, design accessibly and ethically, and use AI with judgment.

**Positioning statement:**
Thoughtful Design teaches designers to move from making interfaces to shaping systems.

**Course promise:**
By the end of the course, learners should be able to look at a complex product problem and ask better questions, identify the underlying system, understand the people affected, make more intentional design decisions, and explain those decisions clearly.

# 2. Learning Philosophy

The course should not feel like a textbook, checklist, or slide deck. It should feel like a guided intellectual journey.

Each module should use a **typographic storytelling framework** that combines narrative, explanation, examples, reflection, and assessment.

The course should teach through:

1. **Story**

   * Start with a realistic product or team situation.
   * Create tension around a familiar design problem.
   * Reveal the deeper design principle through that situation.

2. **Concept**

   * Introduce the relevant framework, method, or mental model.
   * Explain it clearly and practically.
   * Connect it to modern UX work.

3. **Example**

   * Show how the concept appears in real product work.
   * Include before/after comparisons where useful.
   * Use concrete UX scenarios rather than abstract theory alone.

4. **Question**

   * Interrupt the learner with inline reflection prompts.
   * Ask them to predict, diagnose, choose, or critique.
   * Use questions to make learning active.

5. **Application**

   * Give learners small, self-contained exercises.
   * Exercises should not require live critique, group workshops, or capstone projects.
   * Learners should be able to complete them independently.

6. **Assessment**

   * Each module should end with a formal assessment.
   * Assessments should measure comprehension, application, judgment, and transfer.

# 3. Instructional Design Requirements

## 3.1 Module Structure

Each module should follow a consistent structure:

### 1. Opening Scenario

A short narrative that presents a realistic UX problem.

Example:

> A product team asks you to redesign a dashboard. The request seems simple: make the dashboard easier to use. But the deeper problem is not the dashboard. It is that no one agrees what the dashboard is for, which objects matter, who uses the data, or what decisions the dashboard should support.

The opening scenario should create a reason to care about the module.

### 2. Core Claim

Each module should introduce one memorable claim.

Example:

> A screen is only the visible surface of a deeper system.

### 3. Conceptual Explanation

The module should explain the core concept in plain, vivid language.

The explanation should be text-forward, but broken up with strong typography:

* short paragraphs,
* section headers,
* pull quotes,
* contrast tables,
* callout boxes,
* “before / after” moments,
* short examples,
* reflection prompts.

### 4. Inline Knowledge Checks

Each module should include several inline questions.

Question types may include:

* multiple choice,
* multiple select,
* true / false,
* scenario-based judgment,
* ordering,
* matching,
* short written reflection,
* “spot the issue” critique,
* “choose the better rationale.”

Inline questions should appear throughout the module, not only at the end.

### 5. Guided Practice

Each module should include at least one self-led exercise.

The exercise should:

* be small enough to complete independently,
* have clear instructions,
* include an example answer or rubric,
* connect directly to the module’s learning objectives.

### 6. Summary

Each module should end with a short synthesis:

* what changed in the learner’s thinking,
* key terms,
* key takeaways,
* how this module connects to the next one.

### 7. Assessment

Each module should end with a scored assessment.

Assessments should test:

* recall of key concepts,
* interpretation of scenarios,
* application of frameworks,
* design judgment.

# 4. Typographic Storytelling Requirements

The course should use typography as part of the learning experience.

Each module should include recurring content patterns such as:

## 4.1 The Situation

A realistic product or team moment.

Purpose: Ground the learner in a practical scenario.

## 4.2 The Hidden Problem

A short explanation of what is really going on beneath the surface.

Purpose: Teach learners to look past the obvious request.

## 4.3 The Shift

A typographic moment that reframes the lesson.

Example:

> The team asked for a better screen.
> The user needed a clearer system.

## 4.4 The Principle

A memorable takeaway.

Example:

> Thoughtful design begins when we understand what exists before deciding what should appear.

## 4.5 Try It

A small activity or reflection.

## 4.6 Check Yourself

A short inline quiz.

## 4.7 Design Judgment Moment

A scenario where more than one answer could be defensible.

Purpose: Assess nuance rather than memorization.

## 4.8 What Good Looks Like

An example answer, rubric, or model response.

## 4.9 Common Trap

A warning about how designers commonly misuse or misunderstand the concept.

## 4.10 Carry Forward

A closing statement that connects the module to the broader course.

# 5. Assessment Requirements

Because the course is self-led and does not include capstone projects, assessment must be distributed throughout the course.

The platform includes **15 distinct assessment objects** in total: 11 module assessments (one per module), 3 milestone assessments (one per course Part), and 1 final cumulative assessment. These are scored, gated checkpoints with an 80% pass threshold.

**Inline checks are not assessments.** Inline checks are low-stakes Question variants embedded in lesson content. They have no pass threshold, produce no cumulative score, and do not gate progression. They exist to sustain engagement and surface immediate feedback. See §5.1 for inline check requirements.

The course should include four levels of assessment.

## 5.1 Inline Checks

Purpose: Keep learners engaged and confirm understanding during instruction.

Format:

* 3–8 inline questions per lesson,
* immediate feedback upon answer selection,
* explanation for correct and incorrect answers,
* low stakes — no cumulative score visible to the learner,
* no "test environment" framing — inline checks should feel conversational and encouraging, not evaluative,
* learners may proceed without answering inline checks (they do not gate lesson completion).

Examples:

* “Which of these is an object?”
* “What is the strongest research question?”
* “Which design choice creates the greatest accessibility risk?”
* “What is the likely second-order effect?”

## 5.2 Module Assessments

Purpose: Confirm that learners understand and can apply the module’s concepts.

Format:

* 8–15 questions per module,
* scored,
* mix of recall, scenario judgment, and application,
* required passing score.

Recommended passing score: **80%**

Learners should be allowed to retry after reviewing feedback.

## 5.3 Cumulative Milestone Assessments

Purpose: Test transfer across related modules.

Recommended milestones:

### Milestone 1: Seeing the System

Covers:

* From Screens to Systems,
* Object-Oriented UX,
* Systems Thinking.

### Milestone 2: Understanding People and Decisions

Covers:

* UX Research,
* Facilitation,
* Product Strategy.

### Milestone 3: Designing Responsibly

Covers:

* Interaction Design,
* Accessibility,
* Ethics,
* AI.

Each milestone should include longer scenario-based questions that require learners to apply multiple concepts together.

## 5.4 Final Assessment

Purpose: Replace the capstone with a self-led, scenario-based exam.

The final assessment should present one complex product scenario and ask learners to analyze it from multiple perspectives.

Recommended structure:

* 25–40 questions,
* scenario-based,
* multiple choice, multiple select, matching, ranking, and short written responses,
* scored automatically where possible,
* rubric-supported where short responses are included.

The final assessment should measure whether learners can:

* identify system context,
* recognize product objects,
* choose appropriate research methods,
* spot accessibility and ethical risks,
* evaluate AI use cases,
* recommend thoughtful design decisions,
* explain tradeoffs.

Recommended passing score: **80–85%**

# 6. Curriculum Outline

## Module 1: From Screens to Systems

### Purpose

Introduce the central shift of the course: modern UX is not only about designing screens. It is about shaping systems.

### Core claim

A screen is only the visible surface of a deeper system.

### Learning objectives

Learners should be able to:

* explain the difference between screen-level and system-level design,
* identify hidden systems behind a product interface,
* describe how UX decisions affect people, teams, data, workflows, and behavior,
* recognize why modern UX requires broader design judgment.

### Key topics

* The evolution of UX
* Interface design vs. product design vs. systems design
* Design as decision-making
* Design as system behavior
* Hidden dependencies behind simple interfaces
* The role of the thoughtful designer

### Typographic story structure

The module should open with a simple redesign request, such as “make this dashboard easier to use,” then reveal that the real problem involves unclear objects, conflicting stakeholder goals, poor data trust, and mismatched user needs.

### Inline question examples

1. Which of the following is a screen-level problem?
2. Which of the following is a system-level problem?
3. A team asks for a new filter. What should a thoughtful designer ask first?
4. What hidden systems might sit behind a search results page?

### Module assessment

Scenario-based quiz where learners distinguish surface requests from deeper system problems.

---

## Module 2: Object-Oriented UX

### Purpose

Teach learners to identify and model the meaningful objects inside a product before designing screens and flows.

### Core claim

Before you design what people do, understand what they are working with.

### Learning objectives

Learners should be able to:

* define objects, attributes, relationships, actions, and states,
* identify core objects in a product domain,
* explain why object modeling improves UX clarity,
* distinguish objects from pages, features, tasks, and UI components,
* create a simple object map from a scenario.

### Key topics

* Object-oriented UX
* Objects vs. actions
* Attributes
* Relationships
* States
* Mental models
* Product models
* Object maps
* Domain clarity
* How OOUX supports IA, navigation, AI, and design systems

### Typographic story structure

The module should begin with a confusing product flow where the team has designed pages before agreeing on the underlying objects. The story should reveal that confusion in the UI often comes from confusion in the model.

### Inline question examples

1. In a school app, which of these is most likely an object: “Assign,” “Assignment,” “Click,” or “Submit”?
2. Which item is an attribute of a “Course”?
3. Which relationship is most likely true: Teacher → Class, or Button → Dashboard?
4. What is the difference between an object and a page?

### Guided practice

Learners receive a short product scenario and identify:

* 5 core objects,
* 3 relationships,
* 3 attributes for one object,
* 2 possible states.

### Module assessment

Learners analyze a messy product description and answer questions about objects, relationships, attributes, and states.

---

## Module 3: Systems Thinking for UX

### Purpose

Help learners understand products as systems of people, rules, incentives, feedback loops, constraints, and consequences.

### Core claim

Every design decision changes behavior somewhere in the system.

### Learning objectives

Learners should be able to:

* define a system in a UX context,
* identify inputs, outputs, feedback loops, constraints, and dependencies,
* distinguish symptoms from structural causes,
* recognize local optimization,
* anticipate second-order effects.

### Key topics

* Systems thinking
* Feedback loops
* Constraints
* Incentives
* Local vs. global optimization
* Second-order consequences
* Service ecosystems
* Cross-functional dependencies
* System boundaries

### Typographic story structure

Open with a product team improving one metric, such as increasing onboarding completion, while accidentally increasing support tickets, user confusion, or downstream churn.

### Inline question examples

1. What is the likely second-order effect of hiding advanced settings?
2. Which option describes a feedback loop?
3. Which team might be affected by this UX change?
4. What is the difference between a symptom and a system cause?

### Guided practice

Learners map a product change across:

* users,
* support,
* data,
* business metrics,
* trust,
* long-term behavior.

### Module assessment

Learners review a feature proposal and identify likely dependencies, feedback loops, and unintended consequences.

## Module 4: Research as Sensemaking

### Purpose

Position UX research as a practical way to reduce uncertainty and make better product decisions.

### Core claim

Research is not just validation. It is how teams learn what reality is asking of them.

### Learning objectives

Learners should be able to:

* distinguish research questions from interview questions,
* identify assumptions and risks,
* choose appropriate research methods,
* recognize weak evidence,
* synthesize observations into insights,
* connect research findings to design decisions.

### Key topics

* Research as decision support
* Assumptions
* Research questions
* Interviewing
* Usability testing
* Contextual inquiry
* Surveys
* Qualitative vs. quantitative evidence
* Synthesis
* Evidence strength
* Research ethics

### Typographic story structure

Open with a team saying, “We need to validate this design,” then reveal that they have not agreed on the riskiest assumption, the decision they need to make, or what evidence would change their mind.

### Inline question examples

1. Which of these is a research question?
2. Which of these is a leading interview question?
3. What method best fits this uncertainty?
4. Which finding is supported by evidence?

### Guided practice

Learners complete an assumption audit:

* What do we believe?
* Why do we believe it?
* What evidence do we have?
* What would we need to learn?
* What decision will the research inform?

### Module assessment

Learners choose research methods and critique sample findings from a product scenario.

# 9. Question and Quiz Requirements

## 9.1 Question Style

Questions should not merely test vocabulary. They should test whether learners can apply concepts in realistic situations.

Avoid questions like:

> What does UX stand for?

Prefer questions like:

> A team asks you to redesign a dashboard because users “do not understand the data.” Which question best helps uncover whether the issue is visual design, object clarity, data trust, or decision context?

### Design Judgment Moment (special question type)

Some questions should have no single correct answer. These are **Design Judgment Moments**: scenario-based questions where multiple answers are defensible depending on the designer’s values, constraints, or context.

Requirements for Design Judgment Moments:

* Do not display a correct/incorrect indicator — every response is a valid position.
* For each option, surface its trade-off rationale (what this choice prioritizes and what it risks).
* Include a synthesis discussion after all options have been revealed (a “What good looks like” reveal).
* Use these questions to assess nuance and professional judgment, not recall.
* Include at least one Design Judgment Moment per module, typically in the module assessment or at a key conceptual fork in a lesson.

## 9.2 Feedback Requirements

Every quiz question should include feedback.

Feedback should explain:

* why the correct answer is correct,
* why tempting wrong answers are incomplete,
* what principle the question is testing.

## 9.3 Difficulty Progression

Questions should become more complex over time.

Early modules:

* identify,
* define,
* distinguish.

Middle modules:

* diagnose,
* choose,
* apply.

Later modules:

* evaluate,
* prioritize,
* justify,
* anticipate consequences.

## 9.4 Scenario Reuse

The course should reuse a small set of recurring product scenarios across modules so learners can see how different lenses reveal different aspects of the same problem.

Recommended recurring scenarios:

1. School analytics platform
2. Healthcare appointment portal
3. Marketplace onboarding flow
4. Internal enterprise dashboard
5. AI writing or recommendation assistant

# 11. Course-Wide Framework

The course should use one recurring framework throughout.

## The Thoughtful Design Loop

### 1. Model

What exists?

### 2. Understand

Who is affected?

### 3. Align

What decision are we making?

### 4. Shape

What should the experience become?

### 5. Test

What did we learn?

### 6. Account

What consequences are we creating?

### 7. Adapt

What should change next?

This loop should appear throughout the course and become the learner’s primary operating model.

Each module should connect back to one or more parts of the loop.

# 12. Voice and Tone Requirements

The course voice should be:

* clear,
* thoughtful,
* direct,
* grounded,
* warm,
* practical,
* precise,
* story-driven,
* intellectually serious without feeling academic.

The course should avoid:

* hype,
* jargon without explanation,
* absolute claims,
* shaming designers for common mistakes,
* overly corporate language,
* vague inspiration,
* tool-first thinking.

The course should favor positive framing.

Instead of:

> Don’t just design screens.

Use:

> Start by understanding the system the screen belongs to.

Instead of:

> AI will not replace designers.

Use:

> AI increases the value of designers who can ask better questions and make better judgments.

Instead of:

> Accessibility is not optional.

Use:

> Accessibility is one way thoughtful design becomes real for more people.

**Module assessment pass rate among active learners (those who have completed 3+ lessons).**

# 15. MVP Scope

The first version of the platform should include the following. All items in this list are confirmed in-scope for MVP.

## Course content

1. Full course introduction
2. 11 fully specified modules (Parts 1–4)
3. 3 milestone assessments (one per Part, covering Parts 1–3)
4. 1 final cumulative assessment
5. Inline knowledge checks throughout each lesson (3–8 per lesson)
6. Self-led exercises with example answers or rubrics
7. 12 downloadable framework templates (reusable resources)
8. Module assessments for all 11 modules (scored, 80% pass threshold)
9. Public lesson preview: 1 lesson per module accessible without authentication (core claim + one example)

## Platform surfaces

10. **Landing page** — standalone marketing surface, fully public, no authentication required. Includes curriculum preview (all 11 modules, previewed but gated), value proposition, learning method overview, tenets/manifesto section, and enrollment CTA.
11. **Dashboard** — authenticated learner home base. Must feature the **Resume CTA** (Continue to next incomplete lesson) as the primary, most prominent element. Includes module progress overview and certificate block.
12. **Progress visualization** — all 11 modules displayed with per-lesson completion state (not started / in progress / complete) and assessment status. Required, not optional.
13. **Blog** — public editorial surface. Index page + individual post pages. Each post ends with an enrollment conversion CTA.
14. **Resources library** — public. 12 framework templates, each with name, description, associated module, format, and download CTA.
15. **Tenets / manifesto section** — appears as a section on the landing page (not a separate page in MVP).
16. **Certificate generation** — issued automatically when all modules and the final assessment are passed. Learner can download as PDF and share to LinkedIn.

## Access model

17. **Open access within the learning zone** — authenticated learners can navigate to any lesson or module without sequential gating. Completion state is tracked and surfaced visually, but does not block access to other content. Assessment unlock is the only gate: module assessments become available when all lessons in that module are marked complete.

## Content authoring

18. Content is managed via MDX files in the codebase. No CMS-style admin UI in MVP. Content changes deploy through code.

## The MVP should not require:

* live workshops or cohort sessions,
* peer review or peer feedback,
* instructor grading,
* capstone projects or portfolio submissions,
* synchronous discussion,
* admin or course-authoring UI,
* team licensing or manager dashboards.

# 16. Future Enhancements

## v2 Vision: The Practice Hub (Concept C)

The long-term platform vision is "The Practice Hub" — a destination designers return to regularly, not just to take a course. In this model, the blog, resources, tenets, and community features become co-equal offerings alongside the course. The Thoughtful Design tenets become the navigational and identity spine of the entire platform.

This vision requires community and belonging features that do not exist in MVP. Explicitly preserve for v2 planning:

* peer discussion spaces and cohort forums,
* member community (even lightweight async),
* "Practice" identity framing across all surfaces,
* curated resource playlists organized by tenet or career stage.

If Concept A's lesson-preview model is adopted (1 public lesson per module, already in MVP), revisit this after 3 months of learner data. If organic conversion remains below 3%, consider shifting to a fuller hybrid B/A model with more public content.

## Feature backlog for future versions

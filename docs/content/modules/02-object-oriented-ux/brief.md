# Module 2 Brief — Object-Oriented UX

> Locked brief. Source: requirements §6, Module 2. Do not invent scope beyond this.

- **Part:** 1 — Seeing the System
- **Module ID:** `02-object-oriented-ux`
- **Core claim:** Before you design what people do, understand what they are working with.
- **Loop step(s):** Model ("What exists?")
- **Recurring scenario:** School analytics platform
- **Resource:** Object Map Template
- **Public preview lesson:** `objects-before-screens`

## Purpose

Teach learners to identify and model the meaningful objects inside a product before designing screens and flows.

## Learning objectives

Learners should be able to:

- define objects, attributes, relationships, actions, and states,
- identify core objects in a product domain,
- explain why object modeling improves UX clarity,
- distinguish objects from pages, features, tasks, and UI components,
- create a simple object map from a scenario.

## Key topics

- Object-oriented UX
- Objects vs. actions
- Attributes
- Relationships
- States
- Mental models
- Product models
- Object maps
- Domain clarity
- How OOUX supports IA, navigation, AI, and design systems

## Opening story structure

Begin with a confusing product flow where the team designed pages before agreeing on the underlying objects. Reveal that confusion in the UI often comes from confusion in the model.

## Inline check seeds

1. In a school app, which of these is most likely an object: "Assign," "Assignment," "Click," or "Submit"?
2. Which item is an attribute of a "Course"?
3. Which relationship is most likely true: Teacher → Class, or Button → Dashboard?
4. What is the difference between an object and a page?

## Guided practice

Learners receive a short product scenario and identify: 5 core objects, 3 relationships, 3 attributes for one object, 2 possible states.

## Module assessment focus

Learners analyze a messy product description and answer questions about objects, relationships, attributes, and states.

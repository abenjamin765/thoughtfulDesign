# Module 3 Brief — Systems Thinking for UX

> Locked brief. Source: requirements §6, Module 3. Do not invent scope beyond this.

- **Part:** 1 — Seeing the System
- **Module ID:** `03-systems-thinking`
- **Core claim:** Every design decision changes behavior somewhere in the system.
- **Loop step(s):** Model, Account
- **Recurring scenario:** School analytics platform
- **Resource:** System Consequence Map
- **Public preview lesson:** `the-onboarding-win-that-backfired`

## Purpose

Help learners understand products as systems of people, rules, incentives, feedback loops, constraints, and consequences.

## Learning objectives

Learners should be able to:

- define a system in a UX context,
- identify inputs, outputs, feedback loops, constraints, and dependencies,
- distinguish symptoms from structural causes,
- recognize local optimization,
- anticipate second-order effects.

## Key topics

- Systems thinking
- Feedback loops
- Constraints
- Incentives
- Local vs. global optimization
- Second-order consequences
- Service ecosystems
- Cross-functional dependencies
- System boundaries

## Opening story structure

Open with a product team improving one metric (e.g. onboarding completion) while accidentally increasing support tickets, user confusion, or downstream churn.

## Inline check seeds

1. What is the likely second-order effect of hiding advanced settings?
2. Which option describes a feedback loop?
3. Which team might be affected by this UX change?
4. What is the difference between a symptom and a system cause?

## Guided practice

Learners map a product change across: users, support, data, business metrics, trust, long-term behavior.

## Module assessment focus

Learners review a feature proposal and identify likely dependencies, feedback loops, and unintended consequences.
